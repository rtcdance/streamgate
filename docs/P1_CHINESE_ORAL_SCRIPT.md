# StreamGate 中文口语版自述

## 90 秒版本

我做了一个 Go 版的 `NFT-gated streaming backend`，想表达的是：我不是单纯做一个 Web3 demo，而是把自己原来做音视频后端的经验，接到一个真实的 Web3 授权场景里。

这个项目里，Web3 主要放在授权层，不是把整个系统都做成链上应用。流程大概是：

1. 服务端先发一个一次性 challenge。
2. 用户用钱包签名。
3. 服务端验签后发 JWT。
4. 用户请求 manifest 时，再去校验 NFT 是否持有。
5. 通过以后发一个短时 playback token。
6. 后续 segment 请求只校验这个 token，不再每次都查链。

这样做的好处是，链上鉴权是可信的，但播放路径不会因为每个分片都查链而变慢。  
另外我还补了转码任务、worker 调度和状态流转，所以这个项目不只是 Web3，也有比较完整的媒体后台味道。

## 30 秒版本

我做了一个 Go 版的 `NFT-gated streaming backend`，核心就是把我原来做音视频后端的经验，和 Web3 的身份授权揉在一起。

这个项目里有几个关键点：

- 用钱包 challenge 做登录
- 用 NFT 持有情况做内容授权
- 保护 HLS 的 manifest
- 给 segment 发短时 playback token
- 再加上转码和 worker 调度，补成一个比较完整的媒体后端

所以它不是一个纯 Web3 demo，而是一个更像真实业务的媒体访问控制系统。

## 1 分钟版本

我想做的是一个更贴近企业场景的转型项目，不是单纯把区块链功能堆进后端。

所以我把 Web3 放在了最合适的位置，也就是“授权层”：

1. 服务端先发一个一次性 challenge。
2. 用户用钱包签名。
3. 服务端验签后发 JWT。
4. 用户请求 manifest 时，再去校验 NFT 是否持有。
5. 如果通过，就发一个短时 playback token。
6. 后续 segment 请求只校验这个 token，不再每次都打链。

这样做的好处是，鉴权还是链上可信的，但播放路径不会因为每个分片都查链而变慢。

## 3 分钟版本

这个项目我最想表达的，其实是我怎么把自己原来擅长的能力，平滑迁移到 Go + Web3 这套栈里。

我以前更熟的是音视频后端，比如：

- 内容分发
- 访问控制
- 任务调度
- worker pool
- 故障处理

所以这次我没有把项目做成一个纯 NFT 或纯合约 demo，而是把 Web3 放进了一个真实的媒体访问场景里。

现在项目里主要有两条主线：

### 第一条：Web3 授权主线

- `auth/challenge` 做一次性钱包登录
- `auth/login` 做签名验签
- `nft/verify` 做链上 NFT 校验
- `streaming/manifest` 做受保护播放入口

这条线的设计重点其实很简单：

- 不信任用户直接传钱包地址
- 不在每个 segment 上重复查链
- 用短时 token 把“强鉴权”和“低延迟播放”拆开

### 第二条：音视频主线

- `transcode/submit` 提交转码任务
- `transcode/status` 查任务状态
- worker 和 transcoder 负责任务执行、重试、取消和健康检查

这部分其实最贴合我原来的背景，因为它本质上还是一个媒体后台系统，只不过是用 Go 重写的。

## 面试时我会重点讲什么

- 这个项目不是为了炫 Web3，而是为了解决受保护内容访问的问题。
- 我把 Web3 用在身份和授权层，而不是让它变成整个系统的全部。
- 我保留了音视频系统最核心的部分：转码、任务调度、worker、缓存、播放路径。
- 我刻意把链上调用放在 manifest 这一步，避免 segment 热路径反复查链。

## 我不会夸大的地方

- 不会说已经有完整 DRM
- 不会说 RPC 高可用已经完全生产化
- 不会说 metadata 端到端已经完全做完
- 不会说转码控制面已经是完整生产版

这些都可以作为后续增强点，但不是当前项目可信度的关键。

## 收尾说法

如果面试官问我为什么做这个项目，我会这么说：

“我不是想做一个纯 Web3 demo，而是想把我原来最熟的音视频后端能力，接到一个真实的 Web3 授权场景里。这样既能证明我会 Go，也能证明我能把链上身份、内容访问控制和媒体后台结合起来。”
